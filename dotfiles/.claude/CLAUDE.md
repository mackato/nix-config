# CLAUDE.md（共通ルール）

すべての開発リポジトリに共通する基本ルール。各リポジトリの `CLAUDE.md` と併せて参照される。

## 基本ルール

- コミュニケーションは日本語で行い、儀礼的な挨拶は省き、簡潔に。
- ユーザーは音声入力を多用するため、文脈に対して不自然・意味不明な単語が現れた場合は、似た音の単語への置換可能性を考慮して解釈する。
  - カタカナ↔英語: 「イグノア」→「ignore」、「クロードMD」→「CLAUDE.md」
  - 同音異義語: 「広告」→「公告」（いずれも「こうこく」）
  - 類音の取り違え: 「報道会社」→「合同会社」（「ほうどう」と「ごうどう」）
- DRY（単一の真実のソース）を徹底し、情報の重複を避ける。同じ事実・定義・設定・ドキュメントを複数箇所に書かない。重複は解釈ずれと更新忘れの温床になるため、一次情報は一箇所に置き、他の場所からは参照・import で済ませる。
  - コードの抽象化（共通化）は DRY と YAGNI を併せて判断する。共通化するのは同じ概念を表す処理に限り、構造が似ているだけで意味が異なる処理は無理に共通化しない（過度な抽象化は YAGNI に反する）。
- YAGNI を徹底し、複雑化やオーバーエンジニアリングを避ける。いま必要なものだけを実装し、仮想的な将来要件のための抽象化・設定項目・フォールバック・防御コードは書かない。タスク範囲外のリファクタや周辺整理も混ぜない。
- ライブラリ・ツール等の技術ドキュメントを参照する際、MCP server（Context7 等）でのドキュメント検索が可能な場合は、Web 検索よりそちらを優先する。
- Web 情報を参照する際は、一次情報（公式サイト・原典等）を最優先し、二次情報は補助的に使う。特に行政手続・法律・公的制度に関する情報は、原則として一次情報（官公庁の公式ページ・法令本文等）のみを根拠とする。

## セキュリティ

- API キー・トークン等の秘匿情報を応答・ログ・コード例に出力しない。マシン固有値や秘密は環境設定（例: `~/.zshrc.local`）に置く。
- 社外秘・個人情報・認証情報を外部サービス（外部 API・LLM・Web 検索等）へ送信する前に、その旨をユーザーへ伝えて確認を取る。
- 削除・上書きなど不可逆な操作の前に対象を確認する。自分が作成していない、または説明と食い違うものは実行せず報告する。

## Git / GitHub

- コミットメッセージには「何を・なぜ変更したか」を簡潔に書く。件名で変更内容を要約し、必要なら本文で背景・理由を補う。実装の詳細（how）はコードに任せ、メッセージには書かない。
- 1 コミットは 1 つの論理的な変更にまとめ、無関係な変更を混ぜない。
- PR 本文には次の 4 項目を含める: (1) ユーザー指示の要点、(2) 変更の概要、(3) レビューのポイント、(4) 検証方法。
- PR のレビューコメントには各スレッドへ個別に返信し、対応可否を 👍 / 👎 のリアクションで示す（一括返信でまとめない）。
