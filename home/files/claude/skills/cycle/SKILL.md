# /cycle

1 タスクを以下の順で最後まで回す。各段を飛ばさない。

## 1. Plan（read-only）
- 着手前に plan を立てる。**選んだ手法が repo に入っている実バージョンで動くか先に検証**し（ツールのバージョン
を確認）、バージョン依存の注意点を承認前に明示する。
- plan 末尾で **2 つを確定**してユーザー承認を取る
  - 実行モード: `branch` か `worktree`（ブランチ/worktree 名は kebab-case で自分で付ける）
  - レビュー依頼先: `codex` / `copilot` / `none`（以降この名前で統一して呼ぶ。途中で混同しない）

## 2. main から離れる
- `main` で実装を始めない。承認後に
  - branch: `git switch -c <name>`
  - worktree: `git worktree add ../<name> -b <name>` してそこへ移る

## 3. 実装
- plan に沿って実装する。

## 4. PR 前のセルフレビュー（必須）
- PR を出す前に自分の差分を批判的に見直す: 競合状態・タイムゾーン/DST・境界値/off-by-one・取りこぼした alias
 やエッジケース・破壊的副作用。
- lint / typecheck / format をローカルで green にしてから出す。

## 5. PR 作成
- 本文に 4 項目を含める: ①ユーザー指示の要点 ②変更の概要 ③レビューのポイント ④検証方法。

## 6. レビュー対応ループ（reviewer が `none` ならスキップ）
- 1 で決めた reviewer にレビューを依頼する。
- 指摘には**スレッドごとに個別返信**する。見送る場合も理由を残す。
- 修正 → push → 再依頼を、全指摘が反映/合理的に見送り済みになるまで繰り返す（上限 5
ラウンド。超えたらユーザーに判断を仰ぐ）。

## 7. マージ
- **ユーザーの明示的なマージ指示を待つ**（自律マージしない）。
- 指示後に squash-merge する。

## 8. 後片付け
- `main` に戻り、作った branch / worktree を削除する（worktree は remove → branch -D）。
